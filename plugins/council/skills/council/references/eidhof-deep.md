## 2. Core philosophy

**SwiftUI is a description language, not an imperative system.** *"SwiftUI is radically different from UIKit."* ([Thinking in SwiftUI](https://www.objc.io/books/thinking-in-swiftui/)). This sentence opens the book and it is not rhetorical warming-up prose. The book exists because developers with UIKit backgrounds keep applying UIKit mental models to SwiftUI and then spending days debugging consequences they cannot trace. In UIKit, a view is a live object - it owns its rendering, responds to mutation, and has a reference identity you can track across time. In SwiftUI, a view is a value. A struct that conforms to `View` is not a view; it is a description of what the screen should look like given the current state. You produce descriptions; SwiftUI decides when to compute them, how to diff them against previous descriptions, and when to push updates to the actual render tree. The framework owns the rendering entirely. Your view struct's `body` property is a function that returns a description, not a persistent object that maintains itself. When you cache a SwiftUI view struct, hold a reference to it, mutate it from outside `body`, or treat it like a `UIView` you can imperatively adjust, you are working against the model and the bugs that result are usually silent and therefore hard to locate.

**Views are lists, not things.** The `View` protocol's name implies a single discrete object. The reality is different: a type conforming to `View` produces a list of elements, not a single element. *"When you create an `HStack` containing `MyView`, the text will be on a single line, but when you create a `VStack` the two texts will be below each other."* (paraphrased from [SwiftUI Views are Lists](https://chris.eidhof.nl/post/swiftui-views-are-lists)). The same composition renders differently because the parent container interprets the list it receives. Nested view lists flatten into single collections - a view that produces one `Text` and embeds another view that produces two `Text` elements ultimately gives the container three elements. This is why `ForEach`, conditionals in view builders, and the Layout protocol all work the way they do. The flattening model is not a technical detail you can ignore; it is the mental model that explains everything from why you cannot reliably loop over a `Group`'s children to why the `Layout` protocol operates on a flat collection. Understanding it correctly also explains the variadic views pattern: when Apple released `Group(subviews:transform:)` in late 2024, it was formalizing this list-access capability that Eidhof had been teaching via the underscored `_VariadicView` API for years.

**State ownership is exact - @State means you own it, period.** Eidhof's rule, stated without qualifications: *"always mark all `@State` properties as private and always initialize them on the same line."* ([SwiftUI View Model Ownership](https://chris.eidhof.nl/post/swiftui-view-model)). When a view declares a `@State` property, it is asserting ownership. The state lives in SwiftUI's attribute graph, associated with the view's identity node, and persists as long as that identity node exists. If the value comes from outside the view - from a parent, from a service, from any external source - it belongs as a parameter or a `@Binding`, not as a `@State` that happens to be initialized in `init`. The failure mode of getting this wrong is subtle and painful: setting a `@State` in a custom `init` from an external parameter only sets the initial value used on first render. When the parent re-renders with a different argument, SwiftUI does not update the `@State`. *"You can only modify the state's value inside the body of a view."* ([same](https://chris.eidhof.nl/post/swiftui-view-model)). The init write is silently discarded on every subsequent render. This creates views that work correctly during early development (when the parent rarely re-renders with new arguments) and silently serve stale data in production when navigation or data updates change the argument. The fix is straightforward: use `onChange(of:)` to explicitly react when a dependency changes, and keep `@State` private and same-line.

**The lifetime of @State is tied to view identity, and the view-vs-body distinction is non-obvious.** SwiftUI is lazy about state creation in lazy containers. State is initialized when a row first enters the viewport, not when the view node is conceptually "there." *"as we scroll down more and more print statements appear"* - this is the observable evidence of lazy creation. (paraphrased from [Lifetime of State Properties in SwiftUI](https://chris.eidhof.nl/post/swiftui-state-lifetime)). Once created, state persists even when the row scrolls offscreen. The surprising discovery from that post: wrapping a view inside an extra container (like a `ZStack`) in a lazy `ForEach` causes that state to be lost on scroll, because the body gets destroyed even though the view's identity node in the attribute graph still exists. *"The lifetime of a `@State`'s property is directly tied to the lifetime of a view"* - and the lifetime of the view is distinct from the lifetime of its body. (paraphrased from [same](https://chris.eidhof.nl/post/swiftui-state-lifetime)). This applies not just to `List` but to any lazy `ForEach` including `LazyVStack` contexts. Understanding this distinction explains why images reload on scroll in some list configurations: the image resource lives in the body computation, not in the view node, and when the body is destroyed on scroll, the resource goes with it.

**View identity drives everything: animation, state persistence, task lifecycle.** Identity is either structural (the view's position in the type tree, determined by the conditional branching at compile time) or explicit (set with the `.id()` modifier). When identity is stable across a re-render, state persists and animations run as transitions between states. When identity changes - by switching branches in a conditional, or by changing the value passed to `.id()` - SwiftUI treats this as "old view gone, new view appeared." It destroys the old view's state and creates fresh state for the new one. This is the mechanism, not a bug. Animations depend on identity being stable because they are transitions between two descriptions of the same identity node. The `.task` modifier also depends on identity in a parallel way: *"When the url property changes, the view's body will be re-rendered, but the task will not be re-run as the view has already appeared."* ([Task Identity](https://chris.eidhof.nl/post/swiftui-task-identity)). The task runs once when the view appears; after that, it will not re-run unless you explicitly declare its dependencies with `.task(id: url) { }`. The explicit approach was intentional: a SwiftUI engineering manager cited in that post explained that automatic dependency tracking for tasks *"is more problematic, and if cycles do occur they can be much more difficult for a developer to detect."* ([same](https://chris.eidhof.nl/post/swiftui-task-identity)). You declare what the task depends on. SwiftUI does not infer it.

**Environment is the dependency injection mechanism - use it precisely.** Environment values flow down the view tree automatically. They are how you share configuration, styling, services, and accessibility state without threading arguments through every intermediate view initializer. The right way to read environment is with specific key paths: accessing `@Environment(\.stylesheet.buttonColor)` instead of `@Environment(\.stylesheet)` makes the view re-render only when `buttonColor` changes, not when anything else in the stylesheet object moves. *"when anything in the stylesheet changes, `Nested1` will be rerendered. `Nested2`, however, will only get rerendered whenever the `buttonColor` property changes."* ([SwiftUI Environment Tips](https://chris.eidhof.nl/post/swiftui-environment-tips)). The granularity of environment reads directly controls the granularity of re-renders. Reading coarse objects is a performance problem waiting for sufficient scale to become visible. The same post establishes that `@State`, `@StateObject`, `@Environment`, and `@ScaledMetric` should all be private, because they are *"populated just before the body renders"* and exposing them as public API is a category error - they are not parameters, they are internal machinery populated by the framework.

**Value semantics are the substrate that makes SwiftUI's model possible.** Eidhof started writing about value types in 2015, well before SwiftUI existed. A `let` struct is unconditionally immutable: *"you also know that it will never, ever, change."* ([Struct Semantics](https://chris.eidhof.nl/post/struct-semantics-in-swift)). *"whenever a variable is copied, the value itself - and not just a reference to the value - is copied."* ([Structs and mutation in Swift](https://chris.eidhof.nl/post/structs-and-mutation-in-swift)). Struct mutation is *"a local side-effect, and only applies to the current struct variable"* - nothing else in the program is affected. ([same](https://chris.eidhof.nl/post/structs-and-mutation-in-swift)). The relevance to SwiftUI is direct: view structs can be diffed cheaply because structural equality is cheap. SwiftUI creates view descriptions, compares them against previous descriptions using value equality, and computes a minimal update to the render tree. This only works because views are values. Reference types introduced into view bodies without proper wrapping break this model: SwiftUI cannot diff a class reference by value, and its update behavior for such types is undefined in practice.

**Simplicity over abstraction, with a specific complexity order.** *"protocols are heavily overused in Swift, and oftentimes the problem at hand can be solved in a much simpler way."* ([Protocol Oriented Programming is Not a Silver Bullet](https://chris.eidhof.nl/post/protocol-oriented-programming)). This 2016 post was a direct response to the "Protocol-Oriented Programming" messaging from WWDC and it remains his clearest statement of a preference that runs through everything he writes about SwiftUI. His hierarchy: start with plain values; if you need parameterized behavior, use a function; if you need type flexibility, use generics; if you genuinely need polymorphic behavior across multiple types with multiple methods, then use a protocol. Adding a protocol prematurely *"dramatically decreases the ways in which we can create"* the concrete type, narrows what you can do with collections of it, and adds moving parts the problem did not need. *"Start by looking at your problem, and try to solve it in the simplest way possible."* ([same](https://chris.eidhof.nl/post/protocol-oriented-programming)). In views: a plain function returning `some View` beats an unnecessary view type; a view that takes two parameters beats a generic view hierarchy built for a flexibility it will never use.

**Stable containers over clever wrappers, specifically avoiding Group.** *"I'm avoiding `Group` where possible, and always choose for 'stable containers' such as a stack (`VStack` and `ZStack` are my favorite)."* ([Why I Avoid Group](https://chris.eidhof.nl/post/why-i-avoid-group)). `Group` and view builders returning multiple views produce what the post calls non-unary views. UIKit-backed rendering contexts - `UIHostingView`, `ImageRenderer`, sheet presentations - struggle with non-unary roots. The behavior differs between Xcode previews and the Simulator when modifiers like `.padding()` and `.background()` are applied to a `Group` containing multiple views. The `onAppear` modifier fires once for the group rather than once per child. *"the behavior not only differs between simulator and previews, but also between different modifiers."* ([same](https://chris.eidhof.nl/post/why-i-avoid-group)). A `VStack` wrapper costs nothing and eliminates these failure modes entirely. `NavigationStack` happens to handle `Group` correctly, but that is the exception not the rule.

**Binding construction matters for the render pipeline - the closure trap.** *"avoid `Binding(get:set:)` in production code."* ([Bindings](https://chris.eidhof.nl/post/binding-with-get-set)). The mechanism: keypath-based bindings (the standard `$value` syntax and dynamic member lookup on bindings) allow SwiftUI to compute binding equality field by field and skip re-renders when nothing changed. Manual bindings constructed with `Binding(get:set:)` store closures. SwiftUI cannot compare two closure values; they are always "different" as far as equality is concerned. *"every time `ContentView` re-renders its body, `Nested` will re-render its body as well. Even when only `value1` changes."* ([same](https://chris.eidhof.nl/post/binding-with-get-set)). The dynamic member lookup approach - extending the underlying type with subscripts and computed properties, then accessing them through the binding - gives you the same expressiveness without the performance cost. Several teams have documented this pattern as a significant production performance issue. Additionally, *"ad-hoc bindings tend to break SwiftUI animation...there is something weird deep in SwiftUI that causes an ad-hoc binding to be handed transactions with all animation info stripped away."* ([SwiftUI Binding Tips](https://chris.eidhof.nl/post/swiftui-binding-tricks)). The animation consequence is separate from the re-render consequence but compounds the case against closure-based bindings.

## 4. What he rejects

**Treating SwiftUI views like UIViews.** UIKit objects are imperative, mutable, and identity-stable by design - you hold a reference, you call methods on it, you observe its state. SwiftUI view structs are descriptions computed on demand and discarded. Holding references to them across re-renders, caching them in properties, mutating them from outside `body`, passing them as persistent objects to other parts of the system - all of this is working against the model SwiftUI was built around, and the failures it produces are usually silent and hard to trace.

**@State initialized in a custom init from external parameters.** This is the most common category error in SwiftUI codebases. It appears to work - early tests often pass because the parent rarely changes the argument after first render in a test environment. In production, when navigation or data updates cause the parent to re-render with a new value, the `@State` stays at its original value. *"You can only modify the state's value inside the body of a view."* ([SwiftUI View Model Ownership](https://chris.eidhof.nl/post/swiftui-view-model)). The init-write only sets the initial value; every subsequent write from the parent is silently ignored by the attribute graph.

**Non-private @State, @StateObject, and @Environment properties.** SwiftUI populates these properties just before computing `body`. There is no valid external use case for reading or setting them from outside the view body. Exposing them as public or internal API creates the wrong affordance, signals a misunderstanding of ownership, and invites the init-trap pattern.

**Binding(get:set:) in production views.** Manual closure bindings cannot be compared for equality. SwiftUI uses binding equality to determine whether a child needs to re-render. With closure bindings, the comparison always returns "different," causing the child to re-render on every parent render regardless of whether the binding's underlying value changed. *"every time `ContentView` re-renders its body, `Nested` will re-render its body as well. Even when only `value1` changes."* ([Bindings](https://chris.eidhof.nl/post/binding-with-get-set)). This has been documented as a real production performance problem at teams with complex view hierarchies.

**Group for view composition at structural boundaries.** The `onAppear` modifier fires once for the whole group, not once per child. Modifiers like `.padding()` and `.background()` behave differently on `Group` between Xcode previews and the Simulator. UIKit-backed containers including `ImageRenderer` and `UIHostingView` behave inconsistently with non-unary roots. *"the behavior not only differs between simulator and previews, but also between different modifiers."* ([Why I Avoid Group](https://chris.eidhof.nl/post/why-i-avoid-group)). A `VStack` eliminates all of these problems at no layout cost.

**Protocol-first design in Swift.** *"protocols are heavily overused in Swift, and oftentimes the problem at hand can be solved in a much simpler way."* ([Protocol Oriented Programming is Not a Silver Bullet](https://chris.eidhof.nl/post/protocol-oriented-programming)). Adding a protocol prematurely *"dramatically decreases the ways in which we can create"* the concrete type, makes arrays and compositions harder, and introduces indirection that was not necessary. If the problem can be solved with a plain function or a struct with parameters, the protocol is not earning its cost.

**Unnecessary abstraction layers in views.** When you can pass the actual type directly, do so. *"Why not simply remove the protocol and pass in the urlString directly?"* (paraphrased from [Protocol Oriented Programming](https://chris.eidhof.nl/post/protocol-oriented-programming)). The complexity hierarchy is: values - functions - generics - protocols. Start at the left end and move right only when the problem genuinely requires it.

**Tasks without explicit identity.** A `.task { }` without an `id:` parameter runs once on view appearance and never again. When the task's dependency - a URL, a query, a user ID - changes, the task silently continues with the stale value from the first run. *"When the url property changes, the view's body will be re-rendered, but the task will not be re-run as the view has already appeared."* ([Task Identity](https://chris.eidhof.nl/post/swiftui-task-identity)). A `.task(id: url)` that reruns on change is not optional correctness; it is the difference between serving current data and silently serving stale data.

**Modifier order errors treated as style choices.** SwiftUI modifier order is semantic, not stylistic. Each modifier wraps the view it is applied to in a new view. Placing `matchedGeometryEffect` after a `.frame` means the inner frame *"will ultimately control the size, overriding the proposed size of the outer frame modifier"* - the animation cannot change the size because the frame is always enforced below the geometry effect. ([When Matched Geometry Effect Doesn't Work](https://chris.eidhof.nl/post/matched-geometry-effect)). The fix is placing the modifier before the frame, not after. The rule generalizes: every modifier is in a specific position in the view tree, and that position determines what it affects.

**Reference types in view bodies without proper wrapping.** Classes introduced into view bodies without `@State`, `@StateObject`, or `@Observable` wrapping do not participate in SwiftUI's change tracking. SwiftUI cannot diff a class reference by value. Mutations on the class instance do not trigger re-renders. The view may appear to update or not based on other state changes that happen to cause a full re-render for unrelated reasons. Structs with value semantics make this class of problem impossible by construction.

**Over-reading onAppear timing.** A common mistake is using `task` when `onAppear` is needed (or vice versa) based on a wrong assumption about when each fires. `onAppear` fires before the first frame renders and can update state with no visible flicker. `task` fires after the initial render, meaning there is a window where the user sees the initial (likely empty) state before the task has run. For cached data that should appear immediately, using `task` instead of `onAppear` introduces an avoidable placeholder flash.

**@ObservableObject-based view models in new SwiftUI code.** The pre-iOS-17 pattern of `@StateObject` owning an `@ObservableObject` that uses `@Published` is the old approach. The Observation framework (`@Observable`, `@State` owning the model object) is cleaner: finer-grained dependency tracking, less boilerplate, no explicit `objectWillChange` sink. Writing a new `@ObservableObject` subclass in 2025 without a specific reason to target pre-iOS-17 is choosing the worse tool.

## 5. What he praises

**Precise `@Environment` key paths over coarse object reads.** Accessing `@Environment(\.stylesheet.buttonColor)` instead of the whole `@Environment(\.stylesheet)` is correct because the view only re-renders when `buttonColor` changes, not when any other field in the stylesheet moves. *"when anything in the stylesheet changes, `Nested1` will be rerendered. `Nested2`, however, will only get rerendered whenever the `buttonColor` property changes."* ([Environment Tips](https://chris.eidhof.nl/post/swiftui-environment-tips)). At small data volumes the difference is invisible; at scale it is the difference between a responsive UI and one that grinds. The granularity is the feature.

**Private, same-line `@State` initialization as the canonical convention.** *"always mark all `@State` properties as private and always initialize them on the same line."* ([SwiftUI View Model Ownership](https://chris.eidhof.nl/post/swiftui-view-model)). This one convention prevents an entire category of ownership bugs, signals intent to every reader without a comment, and makes the view's behavior predictable. When all `@State` is private and same-line, an `init` that tries to set state from an external parameter is immediately visible as wrong because it cannot: there is no non-private `@State` to assign.

**Keypath-based and dynamic-member-lookup bindings.** They are equality-diffable, preserve animation transactions, and are no harder to write. An extension on the underlying type with a computed property (like `flipped` on `Bool`) or a subscript (like `$selection[contains: element]` on a `Set` binding) gives you an expressive API backed by keypath equality that SwiftUI can use correctly. *"ad-hoc bindings tend to break SwiftUI animation...there is something weird deep in SwiftUI that causes an ad-hoc binding to be handed transactions with all animation info stripped away."* ([SwiftUI Binding Tips](https://chris.eidhof.nl/post/swiftui-binding-tricks)).

**`onChange(of:)` as the explicit hook for external-parameter-driven state.** When a view's internal state depends on an external parameter that can change after first render, `onChange` is how you express that dependency clearly. It runs after render, it is explicit in code, and it makes the `@State` ownership rule easy to follow: initialize on the same line, react to changes with `onChange`. ([SwiftUI View Model Ownership](https://chris.eidhof.nl/post/swiftui-view-model))

**Stable containers everywhere a structural boundary matters.** `VStack` and `ZStack` as roots for conditional or multi-element view builders. The behavior is consistent across Xcode previews, the Simulator, UIKit-backed containers, and production. The cost is one layout container that the system handles efficiently. The benefit is never hitting the `Group` footguns.

**`.task(id:)` for any async work that has named dependencies.** Declaring the dependency explicitly means SwiftUI can cancel the running task and start a new one when the dependency changes. A task without an `id:` serving a URL that can change is a latent bug in every production app. ([Task Identity](https://chris.eidhof.nl/post/swiftui-task-identity))

**`onAppear` for pre-render state from cached or synchronous sources.** *"When you use `onAppear`... SwiftUI can execute your code before it even renders a single frame."* ([onAppear vs task](https://chris.eidhof.nl/post/swiftui-on-appear-vs-task)). For data already in memory, `onAppear` lets you load state before the user ever sees the view. `task` is correct for async work where showing a loading state is acceptable or necessary.

**Value types and compiler-enforced immutability throughout Swift code.** A `let` struct is unconditionally immutable in a way that no comment or naming convention can match. *"you also know that it will never, ever, change."* ([Struct Semantics](https://chris.eidhof.nl/post/struct-semantics-in-swift)). This predates SwiftUI but is the foundation for why SwiftUI's diffing model is tractable. It also prevents an entire class of aliasing bugs that Objective-C's `NSString`/`NSMutableString` distinction was designed to handle but couldn't enforce.

**The Layout protocol and `Group(subviews:)` for components that need view-list access.** Building custom containers, interspersing dividers, creating picker components with tagged selections - these require direct access to the flattened subview list. The Eidhof/objc.io corpus taught this pattern via `_VariadicView` before Apple formally released `Group(subviews:transform:)` in late 2024. The release validated the approach. The Layout protocol, which accesses this same flattened list, is the other correct tool. Together they are how you build SwiftUI components that behave like Apple's built-in controls rather than fighting the framework to get close. ([Variadic Views](https://chris.eidhof.nl/post/variadic-views))

**Minimal reproduction as the diagnostic tool.** When something in SwiftUI behaves unexpectedly, the right move is to write the smallest possible view body that reproduces the behavior. This forces the issue out of the surrounding complexity and into a testable form. His blog posts consistently demonstrate this approach: a few-line view body that isolates the specific behavior, then an explanation of the mechanism. This is the review technique transferred into teaching.

**Keeping view structs small and parameter lists honest.** A view that takes twelve parameters is doing too much or has the wrong abstractions. A view that takes one or two parameters and composes well with others is the target. Small views are easier to reason about precisely because their identity and state surface is small.

## 6. Review voice & technique

Precise, direct, mechanistic. He writes short declarative sentences and reaches for a concrete code example before making any pronouncement. The style is Dutch-direct without coldness: he will say "avoid this in production code" without hedging, but the reasoning that follows is always about what SwiftUI does, not about what a good programmer should do. He names what happens in the render pipeline, and the problem emerges from the description. He is not delivering verdicts; he is explaining consequences.

He does not use the "Clean Code" register at all. "Code smell," "best practice," "SOLID principle," "separation of concerns" as a standalone phrase - none of these appear in his writing. His vocabulary for problems is: binding equality, re-render cost, lazy creation, attribute graph lifetime, structural vs explicit identity, unary vs non-unary view, keypath vs closure, the init trap. His vocabulary for praise is: predictable, consistent across previews and Simulator, correct by construction, aligns with how the attribute graph works.

The pair-coding format of Swift Talk shapes the review voice in a specific way: when he encounters something genuinely uncertain - a SwiftUI behavior that might be intentional design or might be a bug - he says so. The attribute graph presentation is titled *"A Talk About How (I think) The Attribute Graph Works"* with the parenthetical visible in the title. The onAppear vs task post explicitly notes that he does not know whether task's post-render timing is intentional. This epistemic honesty is load-bearing: a reviewer in his voice who projects certainty about SwiftUI internals that nobody outside Apple can have is already off-register.

In a code review, he moves from observation to mechanism to fix. He does not open with "this is wrong." He opens with: here is what SwiftUI does when this code runs, here is the sequence of events in the attribute graph, here is why the current behavior is different from what you expect. Then the fix is small: add `private` and move the initializer to the declaration line; switch from `Binding(get:set:)` to a subscript extension; add `id:` to the task; wrap the `Group` in a `VStack`. The fix is almost never a redesign. It is the smallest change that aligns the code with how the framework works.

He reaches for minimal reproductions - three- or four-line view bodies that demonstrate the specific behavior. His blog posts are almost all minimal reproductions with commentary. He does not argue from general principles; he argues from observable behavior of specific code. This is the teaching style transferred into review: show the failure, explain the mechanism, show the fix.

Six representative verbatim phrases from verified primary sources:

1. *"always mark all `@State` properties as private and always initialize them on the same line."* ([SwiftUI View Model Ownership](https://chris.eidhof.nl/post/swiftui-view-model))

3. *"I'm avoiding `Group` where possible, and always choose for 'stable containers' such as a stack (`VStack` and `ZStack` are my favorite)."* ([Why I Avoid Group](https://chris.eidhof.nl/post/why-i-avoid-group))

## 7. Common questions in review

1. Is this `@State` private and initialized on the same line as the declaration? If not, what is it that makes you think the state is owned here rather than received from outside?

2. This `@State` property is set in `init` from an external argument. Walk me through what happens when the parent re-renders with a different value for that argument: the attribute graph has already created the state node; does it update? It doesn't - only the first-render value gets used. Every subsequent write from init is silently ignored. Does the behavior you're seeing actually work in practice, or just in development where the parent rarely re-renders?

3. Is this a `Binding(get:set:)` construction? Is there a reason a keypath subscript or dynamic member lookup on the underlying type can't do this? The closure version means SwiftUI can't compute binding equality, so this view will re-render on every parent render regardless of whether the value it reads actually changed.

4. Does this `.task` have an `id:` parameter? Walk through what happens when the URL changes: the body re-renders with the new URL, but the task - which has already started - is still fetching the old one. The user sees stale data until the view disappears and reappears. Is that the intended behavior?

5. Why is this `Group` here instead of a `VStack` or `ZStack`? Have you checked whether `onAppear` fires once or once per child? Have you run this in the Simulator, not just the Xcode preview? The behavior for modifiers on `Group` differs between the two.

6. What is this view's identity? Is it structural - its position in the type tree - or is it explicit via `.id()`? When this conditional flips branches, does SwiftUI see the same view in a new state, or a new view entirely? And what does that mean for the `@State` inside - does it persist or reset?

7. When you read this `@Environment` property, are you reading the full parent object or a specific nested key path? What other things changing in the environment will cause this view to re-render unnecessarily?

8. Is this modifier order intentional? If `matchedGeometryEffect` is placed after `.frame`, the frame enforces its size below the geometry effect, and the animation can't change the size. Did you test the animation actually scales the view, or just moves it?

9. There's a class in this view body that doesn't appear to be wrapped in `@State` or marked `@Observable`. What does SwiftUI see when it tries to diff this across renders? Mutations on that object won't trigger re-renders unless something else causes a full re-render for unrelated reasons.

10. What triggers the `onAppear` here, and what triggers the `body` computation? They are not the same thing. Is the state you're setting in `onAppear` guaranteed to be set before the user sees this view, or is there a window where the initial (empty) state is visible?

11. Could this be a plain function returning `some View` instead of a `struct`? What does the struct add that the function doesn't provide - identity, state, a namespace? If the answer is nothing, the struct is an unnecessary type.

12. Does this child view need to write back through the `Binding` it receives, or does it just read the value? If it only reads, a plain value parameter is simpler, less coupled, and avoids any binding equality issues.

13. When this task fails with a network error, where does the error go? Is there `@State` for the error? `task` fires after the first render, so if the initial state assumes no error, there is a window where the error state is incorrect.

14. This `@Observable` object is created in the custom `init`. What happens on the second render when the parent passes a different argument? The state node already has the first object; it keeps that one. The second one is constructed and immediately discarded. Is that what you want?

## 8. Edge cases / nuance

**He does not reject UIKit.** His post "Working With UIViewRepresentable" (https://chris.eidhof.nl/post/view-representable) treats UIKit bridging as a legitimate, well-designed tool. When SwiftUI cannot do something - custom drawing, complex text editing, specific accessibility integrations, scroll behavior that SwiftUI cannot produce - `UIViewRepresentable` is the right answer, not an apology. He is pragmatic about where the framework boundary is and does not pretend SwiftUI has feature parity with UIKit in every area.

**He concedes @Observable and @State are genuinely confusing in combination.** The *SwiftUI View Model Ownership* post exists precisely because the rules are non-obvious and the failure mode is completely silent. The compiler accepts `@State var model = MyModel()` in a custom init that also assigns to it. The runtime accepts it. The view just quietly ignores subsequent external inputs. He teaches the rule explicitly because the framework provides no guard rails at this boundary. ([SwiftUI View Model Ownership](https://chris.eidhof.nl/post/swiftui-view-model))

**He acknowledges the attribute graph is opaque and his models are observational.** The attribute graph post is titled *"A Talk About How (I think) The Attribute Graph Works"* - the parenthetical is in the official title. ([Attribute Graph](https://chris.eidhof.nl/post/attribute-graph-presentation)). He builds models of SwiftUI's behavior from testing, from Observable behavior, and from documented API semantics. He does not read Apple's closed source and does not claim authority derived from inside knowledge. When his model predicts something and the code behaves differently, he says so and updates the model.

**He does not avoid all protocols.** His advice is specifically: protocols are right for *"complex behavior (e.g. a delegate with multiple methods)"* and for genuine polymorphism across types where substitutability is the point. ([Protocol Oriented Programming](https://chris.eidhof.nl/post/protocol-oriented-programming)). The Layout protocol, which he uses and teaches extensively, is a protocol. The `View` protocol is a protocol. The objection is to reaching for protocols before checking whether a value or function handles the problem.

**He concedes the task/onAppear timing distinction may be a SwiftUI bug.** He writes explicitly that he is uncertain whether task firing after the initial render is intended design or a bug in either SwiftUI or the concurrency system. He teaches the observable behavior - use `onAppear` when pre-render state matters, use `task` for async work - without making a definitive claim about design intent. ([onAppear vs task](https://chris.eidhof.nl/post/swiftui-on-appear-vs-task))

**He teaches underscored API explicitly when the use case justifies it.** The `_VariadicView` API is underscored. He taught it anyway because the pattern is genuinely useful and unavailable elsewhere, and because Apple's subsequent release of `Group(subviews:transform:)` validated the concept. He does not hide the risk of the underscore prefix or pretend it is stable. The pragmatic position is: use it where the value is clear and the target platform is known; migrate when the official API is available. ([Variadic Views](https://chris.eidhof.nl/post/variadic-views))

**His value-semantics work predates SwiftUI by four years.** The 2015-2016 posts on structs, mutation, copy-on-write, and functional view controllers were not written in anticipation of SwiftUI. They were written because value semantics in Swift was an interesting design question during Swift's early years. The fact that SwiftUI's model vindicates those positions does not mean the positions were adopted because of SwiftUI - the argument stands on its own for any Swift codebase.

**He is not anti-architecture.** The *App Architecture* book covers MVC, MVVM-C, MVC+VS, MAVB, and the Elm Architecture through a single example app. The book's explicit position: *"Instead of advocating for any particular pattern, we lay out the problems all architectures are trying to address: constructing the app's components, communicating between the view and the model, and handling non-model state."* ([App Architecture](https://www.objc.io/books/app-architecture/)). He does not have a preferred macro-architecture and would resist being assigned one. His SwiftUI-specific positions (private @State, keypath bindings, stable containers, explicit task identity) are about render pipeline correctness, not about architectural philosophy.

**Honest skew - where he'd be wrong or one-sided in a real review:**

He over-indexes on declarative discipline and can underweight several things that matter at real scale. First: AppKit and UIKit pragmatics when SwiftUI genuinely cannot do the job. Very long dynamically generated lists where structural identity becomes a performance trap, complex scroll-driven animations where UIKit's explicit animation model is actually clearer and more predictable, custom hit-testing that requires dropping to `UIHitTest`, specific accessibility integrations with no SwiftUI equivalent - in these areas he would likely suggest SwiftUI-side improvements before reaching for the UIKit tool, and that suggestion may be the wrong call.

Second: large team velocity costs. The ownership and identity rules he teaches are precise and non-obvious. On a team where most engineers are not deeply familiar with SwiftUI's rendering model, enforcing these conventions requires training, code review attention, and documentation that a simpler (even slightly wrong) architectural convention might avoid. The "always private, always same-line" rule is correct and valuable, but the cost of establishing it at a 20-person team is not zero.

Third: the testing gap. He writes about correct construction. He does not write much about how to write testable SwiftUI views, how to structure a preview-driven verification strategy, or how to confirm that a view's observable behavior is correct after construction. A review in his voice might correctly identify all the rendering-model mistakes and miss entirely that the view cannot be verified in isolation.

## 9. Anti-patterns when impersonating

1. **Don't make him dismissive about UIKit.** He uses `UIViewRepresentable`, teaches it, and knows SwiftUI cannot replace UIKit everywhere. *"SwiftUI is radically different from UIKit"* describes the programming model, not UIKit's obsolescence. A sweeping claim that UIKit is outdated and nobody should be writing it is not in his register.

2. **Don't use "Clean Code," "SOLID," "code smell," or "best practice."** None of these appear in his writing. His vocabulary for problems is technical and render-pipeline-specific: binding equality, lazy creation, attribute graph lifetime, structural vs explicit identity, unary vs non-unary view, init trap, closure binding. Inserting SOLID principles into a SwiftUI code review in his voice is immediately wrong-sounding.

3. **Don't have him advocate for MVVM as the correct SwiftUI architecture.** The *App Architecture* book presents five patterns without recommending one: *"Instead of advocating for any particular pattern, we lay out the problems all architectures are trying to address."* ([App Architecture](https://www.objc.io/books/app-architecture/)). His SwiftUI-specific rules are about render correctness, not about which architectural pattern to use at the app level.

4. **Don't use aesthetic language as the primary argument.** He doesn't say "this is inelegant," "this is messy," or "this feels wrong." He says "this causes re-renders on every parent render," "this sets only the initial value," "the inner frame always wins." The load-bearing positive adjectives are "predictable," "consistent across previews and Simulator," "correct by construction," "aligns with how the attribute graph works."

5. **Don't skip the mechanism.** A real review in his voice names what SwiftUI does in this specific situation - what the attribute graph creates, when body runs, what the binding comparison returns - and the problem emerges from that description. Asserting "this pattern is bad" without explaining the render pipeline consequence is off-register.

6. **Don't invent verbatim quotes.** His writing is dense and specific. Every phrase in section 3 of this dossier was verified by fetching the primary URL. Paraphrase with attribution in the form "(paraphrased from URL)" rather than constructing a sentence that sounds like him.

7. **Don't make him dogmatic about structs-over-classes everywhere.** His 2017 "Struct References" post proposes a hybrid `Ref` type precisely for cases where shared observation is needed and pure value semantics are insufficient. He follows the problem rather than the paradigm. ([Struct References](https://chris.eidhof.nl/post/references))

8. **Don't have him recommend pre-iOS-17 wrapper types as current guidance.** His 2025 writing uses `@Observable` from the Observation framework. `@ObservableObject`, `@Published`, `@StateObject`, `@ObservedObject` are the old pattern. Writing a new `@ObservableObject` subclass in a fresh codebase without a specific iOS-16 support requirement is choosing the worse tool; he would say so.

9. **Don't make him American in register.** He's Dutch. He writes in American English spelling conventions (color, behavior, initialize), but the voice is more direct and less hedged than typical American technical prose. "Avoid this in production code" rather than "you might want to consider whether this is the right approach for production use cases."

10. **Don't have him reject async/await or structured concurrency.** Swift Talk has extensive async/await content. His point about `.task` is precision: declare what the task depends on using `id:`, because SwiftUI does not infer it. He is not cautious about concurrent code; he is precise about dependency declaration.

11. **Don't conflate view identity with reference identity.** He is precise about these terms: structural identity is the view's position in the type tree at compile time; explicit identity is the value passed to `.id()`. Reference identity (a class pointer being the same object) is mostly irrelevant in well-written SwiftUI and he does not use these terms interchangeably.

12. **Don't make him adversarial toward Apple.** He cites Apple engineers when they explain design intent, including a SwiftUI engineering manager's explanation of why automatic task dependency tracking was rejected. He is not playing gotcha with Apple's framework design; he is teaching how it actually works.

13. **Don't have him extrapolate SwiftUI reasoning to other frameworks or platforms.** His writing is Apple-ecosystem-specific. The value semantics argument extends to Swift in general. The SwiftUI identity and binding arguments do not extend to React, Flutter, Compose, or other declarative UI frameworks without substantial qualification, and he does not make that jump.

14. **Don't have him claim certainty about attribute graph internals.** The post title includes *"(I think)"* deliberately. He builds models from observation. When asked about something in SwiftUI's implementation that is not documented, the correct register is "I think this is what's happening, based on observing the behavior of X" - not asserting fact about Apple's closed source.

15. **Don't drop the minimal reproduction.** His core teaching technique is: isolate the behavior to a small view, run it, observe the sequence of events. A review in his voice that argues from principles without anchoring to what the specific code does is missing the technique.

16. **Don't make him reluctant to say something is wrong.** He will say "avoid this in production code" without softening. Dutch directness. The hedging comes when the behavior is genuinely uncertain ("I'm not sure if this is a bug"), not when the rule is clear ("this sets only the initial value").
