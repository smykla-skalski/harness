# Runtime config layering

Each agent runtime has two config layers: a project-local file managed by harness and a user-global file owned by the user. They coexist without conflict.

The project-local file wires harness hooks into the runtime. The user-global file holds personal tooling - for example, a user might have `klaudiush` or similar tooling configured there. Harness never touches the user-global layer.

| Runtime  | Project-local (harness-owned)        | User-global (personal)              |
|----------|--------------------------------------|-------------------------------------|
| Claude   | `.claude/settings.json`              | `~/.claude/settings.json`           |
| Codex    | `.codex/hooks.json`, `.codex/config.toml` | `~/.codex/config.toml`         |
| Gemini   | `.gemini/settings.json`              | `~/.gemini/settings.json`           |
| Copilot  | `.github/hooks/harness.json`         | VS Code / Copilot user settings     |
| Vibe     | `.vibe/hooks.json`                   | `~/.vibe/settings.json`             |
| OpenCode | `.opencode/hooks.json`               | `~/.opencode/config.json`           |

The project-local files are created and managed by `harness setup bootstrap`. Do not hand-edit them - re-run bootstrap instead. Any changes you make will be overwritten the next time bootstrap runs.

User-global files are never read or written by harness. If a runtime merges both layers (Claude and Gemini do), your personal config applies on top of the harness project config automatically.

Runtime configs are regenerated by `harness setup bootstrap`. Do not hand-edit project-local config files in managed locations; re-run bootstrap instead.
