# Runtime config layering

Each agent runtime has two config layers: a project-local file managed by harness and a user-global file owned by the user. They coexist without conflict.

The project-local file wires harness hooks into the runtime. The user-global file holds personal tooling - for example, a user might have `klaudiush` or similar tooling configured there. Harness never touches the user-global layer.

| Runtime  | Project-local (harness-owned)        | User-global (personal)              |
|----------|--------------------------------------|-------------------------------------|
| Claude   | `.claude/settings.json`              | `~/.claude/settings.json`           |
| Codex    | `.codex/hooks.json`, `.codex/config.toml` | `~/.codex/config.toml`         |
| Gemini   | `.gemini/settings.json`              | `~/.gemini/settings.json`           |
| Copilot  | `.github/hooks/harness.json`         | VS Code / Copilot user settings     |
| Vibe     | `.vibe/hooks.json`                   | `~/.vibe/settings.json`             |
| OpenCode | `.opencode/hooks.json`               | `~/.opencode/config.json`           |

The project-local files are created and managed by `harness setup bootstrap`. Do not hand-edit them - re-run bootstrap instead. Any changes you make will be overwritten the next time bootstrap runs.

User-global files are never read or written by harness. If a runtime merges both layers (Claude and Gemini do), your personal config applies on top of the harness project config automatically.

Runtime configs are regenerated by `harness setup bootstrap`. Do not hand-edit project-local config files in managed locations; re-run bootstrap instead.

## Feature toggles

Two hook families are gated behind feature flags because their underlying features are unfinished and slow every tool call without producing useful guidance. Both default to off; their entries are simply omitted from the regenerated runtime configs.

| Flag | Env var | CLI flag (`bootstrap` and `agents generate`) | Affects in regenerated config |
|------|---------|----------------------------------------------|-------------------------------|
| Suite-lifecycle hooks | `HARNESS_FEATURE_SUITE_HOOKS=1` | `--enable-suite-hooks` | `guard-stop`, `context-agent`, `validate-agent`, `tool-failure` |
| Repo-policy pre-tool hook | `HARNESS_FEATURE_REPO_POLICY=1` | `--enable-repo-policy` | `repo-policy` (raw `cargo`/`xcodebuild` warnings) |

Resolution: the CLI flag, when present, wins over the env var. Truthy env values follow the harness convention (`1`, `true`, `yes`, `on`, case-insensitive). Re-running `harness setup bootstrap` after flipping a flag rewrites the project-local config files with the new hook set, and bootstrap logs an `info!` line per regenerated config naming any omitted hook family so the change is visible without inspecting the file.

Doctor (`harness observe doctor`) compares the on-disk config against the *env-resolved* expected shape — it reads the env vars at doctor invocation time, not the values that were active at bootstrap. If you bootstrapped via `--enable-suite-hooks` (with no env var set) and later run doctor in a fresh shell, doctor will report drift. To avoid that, either keep the env var set in the shell that runs both commands, or accept the drift report and re-run bootstrap before doctor.
