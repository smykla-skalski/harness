# Runtime config layering

Each agent runtime has two config layers: a project-local file managed inside the repo and a user-global file owned by the user. They coexist without conflict.

The project-local file starts with the harness-owned hooks written by `harness setup bootstrap`. If you want the separate aff-owned repo-policy/session-start hooks, run the dedicated manual `aff:*` setup task afterward. The user-global file holds personal tooling - for example, a user might have `klaudiush` or similar tooling configured there. Repo setup tasks never touch the user-global layer.

| Runtime  | Project-local (repo-managed)         | User-global (personal)              |
|----------|--------------------------------------|-------------------------------------|
| Claude   | `.claude/settings.json`              | `~/.claude/settings.json`           |
| Codex    | `.codex/hooks.json`, `.codex/config.toml` | `~/.codex/config.toml`         |
| Gemini   | `.gemini/settings.json`              | `~/.gemini/settings.json`           |
| Copilot  | `.github/hooks/harness.json`         | VS Code / Copilot user settings     |
| Vibe     | `.vibe/hooks.json`                   | `~/.vibe/settings.json`             |
| OpenCode | `.opencode/hooks.json`               | `~/.opencode/config.json`           |

The harness-owned project-local files are created and managed by `mise run setup:bootstrap` (or `mise run setup:agents:generate` for checked-in assets). The aff-owned hook entries are added only when you manually run the corresponding `aff:*` task. Do not hand-edit project-local config files in managed locations - rerun the relevant setup task instead.

User-global files are never read or written by the repo setup tasks. If a runtime merges both layers (Claude and Gemini do), your personal config applies on top of the project-local config automatically.

Runtime configs are regenerated by `mise run setup:bootstrap`; aff-owned hook entries are patched by the manual `aff:*` tasks. Do not hand-edit project-local config files in managed locations; rerun the relevant setup task instead.

## Feature toggles

One harness-owned hook family is gated behind a feature flag because it is unfinished and slows every tool call without producing useful guidance. It defaults to off, and its entries are simply omitted from the regenerated runtime configs.

| Flag | Env var | CLI flag (`bootstrap` and `agents generate`) | Affects in regenerated config |
|------|---------|----------------------------------------------|-------------------------------|
| Suite-lifecycle hooks | `HARNESS_FEATURE_SUITE_HOOKS=1` | `--enable-suite-hooks` | `guard-stop`, `context-agent`, `validate-agent`, `tool-failure` |

Resolution: the CLI flag, when present, wins over the env var. Truthy env values follow the harness convention (`1`, `true`, `yes`, `on`, case-insensitive). Re-running `mise run setup:bootstrap -- --enable-suite-hooks` (or setting the env var) rewrites the project-local config files with the new hook set, and bootstrap logs an `info!` line naming the omitted family so the change is visible without inspecting the file.

Doctor (`harness observe doctor`) compares the on-disk config against the *env-resolved* expected shape â€” it reads the env var at doctor invocation time, not the value that was active at bootstrap. If you bootstrapped via `--enable-suite-hooks` (with no env var set) and later run doctor in a fresh shell, doctor will report drift. To avoid that, either keep the env var set in the shell that runs both commands, or accept the drift report and rerun bootstrap before doctor.
